REQUISITOS TÉCNICOS – REFATORAÇÃO DO AUTHENTICATION + IPINTEL
=============================================================

OBJETIVO GERAL
--------------
Refatorar a aplicação de autenticação existente, mantendo a lógica já funcional,
preservando a compatibilidade com o frontend atual e adicionando uma camada
mais robusta de segurança, auditoria e investigação de acessos baseada em IP.

A solução será composta por dois blocos separados:

1. authentication
   - App Django responsável por autenticação, sessões, recuperação de senha,
     OTP, reset de senha, auditoria e regras de segurança.

2. ipintel
   - Engine externa responsável por investigação e classificação de IPs,
     incluindo GeoIP, ASN, TOR, Proxy, Datacenter, WHOIS, Reverse DNS,
     fallback por API gratuita e risk score.

======================================================================
1. REQUISITOS OBRIGATÓRIOS DO AUTHENTICATION
======================================================================

1.1 Compatibilidade com o frontend
----------------------------------
O endpoint de login DEVE manter exatamente o mesmo formato de resposta já usado
atualmente no frontend, pois os dados são usados no Bearer token, localStorage
e store do frontend.

Formato obrigatório do login:

{
  "access": "jwt-token",
  "refresh": "refresh-token",
  "user": {
    "id": 1,
    "email": "user@email.com",
    "first_name": "",
    "last_name": "",
    "role": "user",
    "is_staff": false,
    "is_superuser": false,
    "is_active": true
  }
}

Não alterar esse contrato.

1.2 Funcionalidades obrigatórias
--------------------------------
O novo authentication deve manter e melhorar tudo que já existe:

- register
- login
- logout
- recovery
- otp verify
- reset password
- endpoint /me/
- block user
- controle de sessões
- auditoria de login
- proteção contra brute force
- envio de e-mails

1.3 Serviço de e-mail
---------------------
O sistema DEVE continuar usando o EmailService já existente no projeto.

Padrão atual a ser mantido:
EmailService.send(template_key=..., to=[...], context={...})

Templates esperados:
- welcome
- login_alert
- security_login_alert
- password_recovery
- password_changed

A lógica de envio de e-mail não deve ser removida.
Deve apenas ser reorganizada para a camada de services.

1.4 Separação de responsabilidades
----------------------------------
A lógica de negócio não deve mais ficar em serializers.

Nova separação obrigatória:

- models       -> persistência
- serializers  -> validação de entrada
- views        -> endpoints
- services     -> regras de negócio
- middleware   -> validações de segurança em request
- dto          -> formato de saída padronizado

1.6 Models obrigatórios
-----------------------
Devem existir e ser usados:

- LoginAttempt
  Controle de tentativas por e-mail + IP.
  Deve permitir bloqueio temporário após falhas sucessivas.

- LoginEvent
  Auditoria completa dos eventos de login, com sucesso/falha, IP, país,
  cidade, ASN, browser, sistema operacional, device e risk_score.

- OtpCode
  Código temporário de recuperação, com validade e marcação de uso.

- ResetPasswordToken
  Token de redefinição de senha com validade e marcação de uso.

- UserSession
  Sessão real vinculada ao JWT, com session_id, token_jti, IP, user agent,
  browser, os, device, country, city, asn, risk_score e flag de sessão ativa.

- UserDevice
  Registro e fingerprint de dispositivos usados pelo usuário.

- SecuritySettings
  Configuração de segurança por usuário, como:
  - max_sessions
  - otp_enabled
  - email_alerts
  - require_otp_for_new_device
  - block_high_risk_login

1.7 Regras de segurança obrigatórias
------------------------------------
- Limite de sessões simultâneas por usuário
- Validação de sessão ativa a cada request autenticado
- Proteção contra brute force
- Registro de auditoria para login com sucesso e falha
- Detecção de login suspeito
- Notificação por e-mail de novo login
- Notificação por e-mail de login suspeito
- Possibilidade de revogar sessões
- Revogação de todas as sessões após reset de senha

1.8 Middleware obrigatório
--------------------------
Criar middleware de validação de sessão autenticada.

Regra:
- Todo JWT deve conter session_id
- O session_id deve existir em UserSession
- A sessão deve estar ativa
- Se a sessão estiver revogada, o token não deve funcionar

1.9 Services obrigatórios
-------------------------
- AuthService
  Login principal, autenticação, auditoria, risco, sessão e token.

- RegisterService
  Cadastro de usuário, associação de grupo e e-mail de boas-vindas.

- RecoveryService / OTPService
  Fluxo de recuperação de senha e envio de OTP.

- PasswordService
  Reset de senha com invalidação do token e revogação de sessões.

- SessionService
  Controle completo de sessões.

- TokenService
  Geração de access/refresh token mantendo o contrato atual.

- LoginAttemptService
  Rate limit e brute force por IP/e-mail.

- LoginAuditService
  Registro de eventos.

- LoginSecurityService
  Avaliação de login suspeito e disparo de alertas.

- DeviceService
  Fingerprint e rastreamento de devices.

- SecuritySettingsService
  Carregamento/criação das configurações do usuário.

- IPIntelligenceService
  Adaptador entre Django e /opt/ipintel.

1.10 DTO obrigatório

- id
- email

- role

- is_active

2.3 Fontes e validadores
------------------------
A investigação de IP deve funcionar com múltiplas camadas:

1. Cache local
2. Base local MaxMind GeoLite2
3. ASN local
4. Lista de TOR exit nodes
5. Lista de proxies públicos
6. Lista de ASN de datacenter/cloud providers
7. Reverse DNS
8. WHOIS
9. API externa gratuita como fallback

2.4 Funcionalidades obrigatórias
--------------------------------
- Identificar país, cidade e coordenadas aproximadas
- Identificar ASN e organização
- Verificar se IP é TOR
- Verificar se IP é proxy
- Verificar se ASN é de datacenter
- Obter reverse DNS
- Obter WHOIS
- Aplicar fallback por API gratuita
- Gerar risk_score
- Registrar cache local
- Ter scripts de atualização via cron

2.5 Risk score obrigatório
--------------------------
A engine deve calcular uma pontuação de risco com base em regras como:

- TOR
- Proxy
- Datacenter
- GeoIP ausente
- Reverse DNS de provedor cloud
- Outras evidências futuras

Resultado esperado:
- risk_score
- risk_reasons

2.6 Uso no authentication
-------------------------
O authentication deve consumir o ipintel por meio do service:
authentication/services/ip_intelligence_service.py

O resultado deve ser usado internamente para:
- auditoria
- risk_score da sessão
- envio de alerta
- detecção de login suspeito

Esses dados NÃO devem quebrar o payload atual do frontend.
